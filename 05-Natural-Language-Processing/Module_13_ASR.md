## 📌 Chapter 1：什么是 ASR？任务定义与难度

### 形式化定义

```
输入：x = [x₁, x₂, ..., xT]  （声音振幅数组）
    ↓
输出：Y = (y₁, y₂, ..., yₘ)  （文字 token 序列）
```

> **本质**：ASR 是一个从一维声学时间序列映射到字符序列的函数。这个函数内部包含了语言、语境、发音的全部复杂性。

### 难度由四个维度决定

| 维度 | 容易 | 困难 |
|---|---|---|
| **词汇量** | 2 词（yes/no） | 60,000+ 词开放对话 |
| **说话场景** | 人对机器说话（会自动放慢） | 两人自然对话（连读、省音） |
| **信道噪音** | 安静室内+头戴麦克风 | 嘈杂街头+远场麦克风 |
| **口音/说话人** | 标准口音成年人 | 方言/儿童/非母语者 |

### 真实 WER 数字（2023-24 年水平）

| 任务 | WER | 难度根源 |
|---|---|---|
| LibriSpeech 有声书（干净）| **1.4%** | 朗读、安静、标准口音 |
| Switchboard 陌生人电话 | 5.8% | 自然对话、电话信道 |
| AMI 会议转写 | 11% | 多人、远场麦克风 |
| CORAAL 非裔美国英语 | 16.2% | 方言、训练数据不足 |
| CHiME6 晚宴四人对话 | **25.5%** | 多人+远场+噪音+重叠 |
| Swahili（低资源语言）| 51.2% | 标注数据极少 |

> **产品启示**：同一个 ASR 系统，用于有声书和嘈杂会议，WER 可以差 20 倍。选型前必须对齐应用场景。

---

### 🎯 产品经理要掌握到：

✅ **ASR 不是一个单一任务**，必须用具体场景对应具体指标：

```
客服电话（电话信道 + 自然对话）→ WER 参考 Switchboard 5-10%
会议记录（多人远场）          → WER 参考 AMI 10-15%
有声书/播客（朗读）           → WER 参考 LibriSpeech 1-3%
```

✅ **训练数据分布 = 使用场景分布**：用有声书数据训练的模型遇到方言/儿童会大幅下降，这是**产品验收时必须测试的场景覆盖问题**

✅ **WER 可以超过 100%**（因为有插入操作），看到 WER > 50% 意味着该场景基本不可用

> **产品面试常问**：
> - "你们的 ASR 产品 WER 是多少？这个数字在什么测试集上测的？"
> - "为什么同一模型在某个用户群体上效果特别差？"（→ 训练数据分布不匹配）

## 📌 Chapter 2：CNN 在 ASR 中的应用

### 为什么用 CNN 而不是全连接层？

```
全连接层（Dense）：每个输出看到全部输入
CNN：每个输出只看局部窗口（几帧）

为什么语音需要局部性？
→ 语音的关键特征（共振峰、爆破音）只在几十毫秒的局部时间窗内存在
→ 说 "dinner" 时，'d' 的声学特征只在那几帧，和前后的静音无关
→ 全连接层引入无关信息，CNN 只看局部更有效
```

**教授比喻**：CNN 的 kernel 就像一个小探照灯，沿时间轴扫过去，每次只照亮一小段，问「这里有我要找的模式吗？」

### 一维卷积（互相关）

**实践中跳过 kernel 翻转步骤**（因为 kernel 参数是训练出来的，翻不翻转不影响学习）

带 Padding 的公式（kernel 宽度 $k=2p+1$，两端各补 $p$ 个零）：

$$z_j = \sum_{i=-p}^{p} x_{j+i} \cdot w_{i+p}$$

**核心直觉**：点积 = 相似度。当 kernel 与局部信号「方向一致」时点积大，不匹配时点积小。kernel 在训练中被学成「检测特定声学模式的探测器」。

### 步幅（Stride）：压缩序列长度

| Stride | 输出长度 | 用途 |
|---|---|---|
| 1 | = 输入长度 | 特征提取，不压缩 |
| 2 | = 输入长度 / 2 | 压缩序列，减少计算量 |

**Whisper 的具体计算（教授演示）**：

```
起点：30 秒音频，stride = 10ms

帧率 = 1秒 ÷ 0.01秒 = 100 帧/秒
总帧数 = 30 × 100 = 3000 帧

第 1 层 Conv1D（stride=1）：3000 帧 → 3000 帧
第 2 层 Conv1D（stride=2）：3000 帧 → 1500 帧（每 20ms 一个 token）
```

**为什么要压缩？** 声学序列比文字序列长 5-19 倍。一个词平均 250ms = 25 帧，但只对应 ~1.3 个 BPE token。不压缩的话 Transformer 的 $O(n^2)$ 计算量会极大。

### 多通道 CNN

- 每帧输入是 **128 维** log mel 谱（不是标量）
- Kernel 形状：`[128, 3]`（深度 × 宽度）
- Whisper 产生 **1280 维**输出 → 需要 1280 个独立 kernel

$$z = \sum_{c=1}^{C_{input}} x^c * w$$

---

### 🎯 产品经理要掌握到：

✅ **CNN 是 ASR 的「前端」**：把原始声音转成结构化特征，这一步的质量直接影响后续所有处理

✅ **理解 stride 压缩的产品含义**：stride 越大，处理速度越快，但可能损失细节——影响产品的**延迟 vs 精度**权衡

---

### 📐 算法工程师要掌握到：

✅ 理解 1D Conv 的 padding/stride 计算（面试可能出现）

✅ 知道 ASR 中用的是**互相关（cross-correlation）**而非严格意义的卷积（省略了翻转 kernel）

## 📌 Chapter 3：编码器-解码器架构（AED / Whisper）

### 解决两大核心问题

```
问题 1：长度不匹配
  声学序列（1500帧）远长于文字序列（几十个字符）
  → 编码器把整段音频压缩成上下文表示

问题 2：对齐未知
  不知道哪些帧对应哪个字母（每次说都不同）
  → 解码器用 Cross-Attention 动态对齐
```

> **教授比喻**：先认真听完并记笔记（Encoder），再一句句复述（Decoder）。

### 完整流程

```
音频波形
  → log mel 谱（128维/帧）
  → CNN 压缩（3000帧 → 1500帧）
  → 正弦位置编码
  → Transformer Encoder → H_enc
  → Transformer Decoder（Cross-Attention）
  → 逐字母生成文字
```

### 编码器 vs 解码器关键区别

| | 编码器 | 解码器 |
|---|---|---|
| **注意力方向** | 双向（可看整个音频） | 单向（只看已生成的词） |
| **任务** | 理解整段音频 | 逐字符生成转写文本 |
| **类比** | BERT | GPT |

### 交叉注意力（Cross-Attention）

$$Q = H^{dec[\ell-1]} W^Q \quad \text{（来自当前解码位置）}$$
$$K = H^{enc} W^K,\quad V = H^{enc} W^V \quad \text{（来自编码器输出）}$$
$$\text{CrossAttention}(Q,K,V) = \text{softmax}\!\left(\frac{QK^\top}{\sqrt{d_k}}\right) V$$

**关键理解**：
- Query = "我现在要生成哪个字母？"
- Key/Value = "整段音频所有帧的信息"
- 注意力权重 = "生成这个字母时，应该重点看音频的哪段"
- **每生成一个字母，都能动态回望整段音频的任意位置**

### 推理策略

**贪心解码**：
$$\hat{y}_i = \arg\max_{\text{char}} P(\text{char} \mid y_1...y_{i-1}, X)$$

**Beam Search + 语言模型重打分**（更高质量）：
$$\text{score}(Y|X) = \frac{1}{|Y|_c} \log P(Y|X) + \lambda \log P_{LM}(Y)$$

分母 $|Y|_c$ 是字符数，防止偏向短句。

### 训练

$$L_{CE} = -\sum_{i=1}^{m} \log p(y_i \mid y_1,...,y_{i-1}, X)$$

配合 **Teacher Forcing**：训练时用真实标签而非预测结果作为下一步输入。

**Whisper 的训练规模**：68 万小时语音（约 77 年），覆盖 96 种语言。

---

### 🎯 产品经理要掌握到：

✅ **Whisper 是离线场景的首选**：精度最高，但需要等说完整句话才能输出，不支持实时字幕

✅ **语言模型融合是提升产品质量的重要手段**：纯 ASR 不懂语言规律，加入 LM 重打分能修正"明显错误"的转写结果

✅ **68 万小时训练数据的商业含义**：这是为什么 Whisper 在多语言场景下表现好——数据覆盖了 96 种语言

> **产品面试常问**：
> - "如果要做会议实时字幕，你会选什么方案？为什么不用 Whisper？"（→ Whisper 不支持流式，需要用 RNN-T 或 CTC）

---

### 📐 算法工程师要掌握到：

✅ 理解 Cross-Attention 是 ASR 中解决对齐问题的核心机制

✅ 知道 Teacher Forcing 带来的 **Exposure Bias** 问题（训练时有真实历史，推理时只有预测历史）

## 📌 Chapter 4：自监督模型 HuBERT

### 核心动机：摆脱对标注数据的依赖

```
编码器-解码器的痛点：
  需要大量【音频 + 文字转写】配对数据
  转写成本极高（需人工逐字标注）

HuBERT 的解法：
  用海量【无标注音频】预训练（播客/YouTube/电话录音）
  只用少量标注数据微调
  → 自监督学习
```

**类比 BERT**：遮住输入的一部分，训练模型预测被遮住的内容。HuBERT 把这个思想用在语音上：遮住部分声学帧，预测它们属于哪个**音素类**。

### 两阶段训练

#### 阶段 1：MFCC → 100 个粗粒度目标

```
Step 1：提取 MFCC（39维声学特征，专门突出音素相关成分）
Step 2：K-means 聚类为 100 个类（这 100 个类 = 训练目标标签）
Step 3：随机掩码部分帧，训练 HuBERT 预测被掩码帧的类别
目的：给模型一个「偏向音素」的弱监督初始化
```

#### 阶段 2：HuBERT 自身表示 → 500 个精细目标

```
Step 1：取第一阶段训练好的模型
Step 2：抽取第 6 层 Transformer 的输出（768维）
Step 3：K-means 聚类为 500 个更精细的类
Step 4：用这 500 个类重新训练
目的：模型学到比 MFCC 更好的表示后，用自己的表示作为目标（自举）
```

> **自举（Bootstrapping）的直觉**：MFCC 给模型一个还不错的起点，模型训练后能学到比 MFCC 更好的声学表示，然后用自己更好的表示作为下一轮的训练标签——越学越好。

### K-means 算法

**初始化**：随机选 k 个向量作为质心 $\mu_1,...,\mu_k$

**迭代（直到收敛）**：

**分配步骤**：
$$\text{cluster}(i) = \arg\min_j \|v^{(i)} - \mu_j\|^2$$

**重估步骤**：
$$\mu_i = \frac{1}{|S_i|} \sum_{v \in S_i} v$$

> 注意：新质心不一定是数据集中的实际点（是均值）

### 微调用于 ASR

```
预训练 HuBERT
    ↓ 去掉投影层和余弦相似度层
    ↓ 加随机初始化的线性 + Softmax 层（29类：26字母+空格+逗号+句号）
    ↓ 冻结 CNN 层（不更新）
    ↓ 用 CTC 损失在标注数据上微调 Transformer
```

---

### 🎯 产品经理要掌握到：

✅ **自监督预训练是解决低资源语言 ASR 的关键技术**：小语种标注数据少，但音频数据相对丰富

✅ **"微调所需数据远少于从头训练"是产品的商业价值**：可以用少量领域数据（如医疗、法律行话）快速适配垂直场景

✅ **HuBERT 的两阶段思路 = 迭代改进**：先用弱标签（MFCC 聚类）学基础，再用更好的表示学精细特征

---

### 📐 算法工程师要掌握到：

✅ 理解 **MFCC** 的作用（39维，专门提取音素相关特征）

✅ 理解两阶段训练的逻辑（弱监督 → 自举精化）

✅ 知道微调时冻结 CNN 层（节省计算，CNN 学到的底层特征已经足够）

## 📌 Chapter 5：CTC（Connectionist Temporal Classification）

### 对齐问题（ASR 的根本难题）

```
说 "dinner" 时：
  'd' 占几帧？'n' 占几帧？'r' 占几帧？
  → 每个人每次说都不同，完全不规则
  → 这就是「对齐问题」

CTC 的策略：
  不找精确对齐，而是「每帧都输出一个字符，再用折叠函数压缩」
```

### 核心机制：blank + 折叠函数 B

引入特殊 **blank 符号（ε）** 解决两个问题：
1. 表示「此帧无需转写」（静音或过渡帧）
2. 分隔重复字母（如 "dinner" 中的双 n）

**折叠函数 B 的规则（顺序不能颠倒）**：
1. 先合并连续重复字符
2. 再删除所有 blank 符号

**转写 "dinner" 的示例**：

```
对齐 A：d  i  ε  n  ε  n  e  r  r  ε  r  r
Step 1：d  i  ε  n  ε  n  e  r     ε  r    （rr → r）
Step 2：d  i     n     n  e  r        r    （删ε）
输出 Y：d  i  n  n  e  r  ✓
```

> **折叠函数是多对一的**：许多不同的对齐都能折叠成同一输出。这是 CTC 的优雅之处——模型不需要学固定对齐，只要学任何能折叠到正确文字的对齐都行。

### 独立性假设：CTC 的优势与代价

$$P_{CTC}(A|X) = \prod_{t=1}^{T} p(a_t|X)$$

给定输入 X，每帧的输出相互独立。

| | 优点 | 缺点 |
|---|---|---|
| **流式识别** | ✅ 不需要看完整输入，支持实时转写 | — |
| **语言建模** | — | ❌ 不知道"q 后面通常跟 u"，每帧独立决策，**必须显式融合外部 LM** |

### 推理

**贪心解码（有缺陷）**：
$$\hat{a}_t = \arg\max_{c \in C} p_t(c \mid X)$$

**问题**：最优单个对齐 ≠ 最优输出字符串！同一输出对应多种对齐，它们概率之和可能更高。

**正确做法**：
$$P_{CTC}(Y|X) = \sum_{A \in B^{-1}(Y)} P(A|X) = \sum_{A \in B^{-1}(Y)} \prod_{t=1}^{T} p(a_t|h_t)$$

实践中用**带合并的 Beam Search** 近似。

### 训练

$$L_{CTC} = -\log P_{CTC}(Y|X) = -\log \sum_{A \in B^{-1}(Y)} \prod_{t=1}^{T} p(a_t|h_t)$$

用**前向后向算法（Forward-Backward）** 动态规划高效计算，与 HMM 训练是同类算法。

融合语言模型：
$$\text{score}_{CTC}(Y|X) = \log P_{CTC}(Y|X) + \lambda_1 \cdot \log P_{LM}(Y) + \lambda_2 \cdot L(Y)$$

### CTC + 编码器-解码器联合训练（现代主流）

**训练时**：
$$L = -\lambda \cdot \log P_{encdec}(Y|X) - (1-\lambda) \cdot \log P_{CTC}(Y|X)$$

**推理时**：
$$\hat{Y} = \arg\max_Y \left[ \lambda \cdot \log P_{encdec} + (1-\lambda) \cdot \log P_{CTC} + \gamma \cdot \log P_{LM}(Y) \right]$$

> **组合优势**：CTC 擅长帧级对齐，编码器-解码器擅长序列建模。现代主流 ASR 系统大多采用联合训练。

### RNN-T（工业级实时转写首选）

CTC 的升级版：去除独立性假设，同时保留流式能力

```
CTC 声学编码器 → h_t^enc
预测网络（Predictor）→ h_u^pred（接收上一个输出 token，充当语言模型）
联合网络（Joint Network）→ 合并两者，预测下一字符
```

$$P_{RNN-T}(Y|X) = \sum_{A \in B^{-1}(Y)} \prod_{t=1}^{T} p(a_t \mid h_t, y_{<u_t})$$

条件包含了输出历史 $y_{<u_t}$，克服了独立性假设 → **手机语音输入、实时字幕的首选**。

---

### 🎯 产品经理要掌握到：

✅ **CTC 的流式能力 = 实时产品的关键**：用户说话的同时就能看到文字，体验完全不同于等完整句子

✅ **CTC 必须加外部语言模型**：纯 CTC 不懂语言规律，实际产品中一定要融合 LM 才能达到可用的质量

✅ **三种架构的产品选型指南**：

```
离线转写（会议记录/字幕生成）→ Whisper（AED），精度最高
实时字幕/语音输入（低延迟）  → RNN-T，工业首选
低资源语言/边缘设备          → CTC + HuBERT，自监督减少标注需求
```

> **产品面试常问**：
> - "为什么 CTC 必须加语言模型？"（→ 独立性假设导致没有语言建模能力）
> - "blank 符号解决了什么问题？"（→ 过渡帧的处理 + 重复字母的区分）

## 📌 Chapter 6：评估 —— 词错误率（WER）

### WER 的定义

$$\text{WER} = 100 \times \frac{S + I + D}{N}$$

| 符号 | 含义 |
|---|---|
| $S$（Substitution） | 替换错误数（识别成了别的词） |
| $I$（Insertion） | 插入错误数（多了不该有的词） |
| $D$（Deletion） | 删除错误数（漏掉了词） |
| $N$ | 参考文本总词数 |

> **重要**：因为有插入操作，**WER 可以超过 100%**！WER 是编辑距离，不是普通百分比。

### 计算示例

**简单例子**：

```
参考（REF）：The  cat  sat  on  the  mat  （N=6）
假设（HYP）：The  cat       on  the  mat  （5词）

对齐：The cat [D:sat] on the mat
S=0, I=0, D=1
WER = 100 × 1/6 = 16.7%
```

**复杂例子（CallHome）**：

```
REF: i *** ** UM the PHONE IS i LEFT THE portable **** PHONE UPSTAIRS last night
HYP: i GOT IT TO the ***** FULLEST i LOVE TO portable FORM OF STORES last night

S=6, I=3, D=1, N=13
WER = 100 × (6+3+1)/13 = 76.9%
```

### 评估前的文本规范化（必须做！）

```
1. 去除元语言标记：[咳嗽]、[笑声]
2. 处理填充词："uh"、"um"——统一处理方式
3. 统一缩略形式："I'm" ↔ "I am"（选一种）
4. 规范化非标准词："$100" → "one hundred dollars"
5. 统一拼写：colour / color
```

> 规范化不统一会人为增加 WER，导致不同系统之间无法公平比较。

### 统计显著性：MAPSSWE 检验

**问题**：系统 A WER=15%，系统 B WER=17%，差异是真实的还是随机波动？

**为什么不用 McNemar 检验？**

McNemar 假设错误相互独立，但语音识别中**相邻词的错误高度相关**（识别错了 "the"，后面词更可能也错）。

**MAPSSWE 的做法**：

1. 把文本分段，边界是**两个系统都识别正确的词**
2. 对每段计算差值：$Z_i = N_A^i - N_B^i$
3. 对所有 $Z_i$ 做 t 检验

当 $n>50$ 时，统计量：
$$W = \frac{\hat{\mu}_z}{\sigma_z / \sqrt{n}}$$

近似正态分布，标准双尾检验（$p < 0.05$）。

### WER 的局限性

| 局限 | 说明 |
|---|---|
| 所有错误权重相同 | 关键词"Boston"错了比功能词"the"错了影响大得多 |
| 不考虑语义相似性 | "automobile"和"car"算同等错误，但语义几乎等价 |
| 对长度敏感 | 短句子一个错误对 WER 影响更大 |

> 尽管有局限，WER 仍是事实标准——计算简单、可复现、跨系统可比较。

### 🎯 产品经理要掌握到：

✅ **WER 是产品验收和 SLA 的核心指标**：谈供应商合同时，要约定在什么测试集上、什么场景下达到什么 WER

✅ **必须问清楚"WER 在哪个测试集上测的"**：在 LibriSpeech 上的 WER 和在你的实际场景上的 WER 可能差 10 倍

✅ **文本规范化不做好，WER 数字没有参考意义**：不同团队的规范化标准不同，数字不可比

> **产品面试常问**：
> - "WER 能超过 100% 吗？为什么？"（→ 可以，插入错误导致）
> - "如何评估你们的 ASR 系统是否比竞品好？"（→ 统一测试集+MAPSSWE 检验）

## 📌 Chapter 7：三大架构横向对比

| 特性 | AED（Whisper） | CTC + HuBERT | RNN-T |
|---|---|---|---|
| **流式识别** | ❌ 需等完整输入 | ✅ 逐帧输出 | ✅ 逐帧输出 |
| **隐式语言模型** | ✅ 解码器学到 | ❌ 独立性假设 | 部分 ✅（Predictor）|
| **精度** | 最高 | 较低（加LM后提升）| 高 |
| **训练数据** | 需大量配对数据 | 可用无标注预训练 | 需配对数据 |
| **计算复杂度** | 高（全局注意力）| 低（逐帧独立）| 中 |
| **典型应用** | 离线转写（字幕/记录）| 低资源/边缘设备 | 工业级实时转写 |
| **外部 LM** | 可选（锦上添花）| 必须 | 可选 |
| **代表系统** | Whisper | HuBERT + fine-tune | Google/Apple 语音输入 |

```
需要实时输出（实时字幕/语音输入）？
    ↓ 是
    → 工业级精度要求？→ 是 → RNN-T
    → 低资源/边缘设备？→ 是 → CTC + HuBERT
    ↓ 否（离线转写）
    → 追求最高精度 → Whisper（AED）
    → 标注数据少 → HuBERT 预训练 + 少量数据微调
```

## ⚡ 踩坑 & 经验

1. **ASR 在演示和生产中的差距极大**：演示时用安静环境+标准口音，生产中遇到各种噪音和方言，WER 可以差 10 倍

2. **CTC 不加语言模型基本不可用**：纯 CTC 的输出经常出现"独立假设导致的荒谬错误"（如 "qu" 中间少了 u）

3. **Teacher Forcing 带来的 Exposure Bias**：训练时有真实历史，推理时只有预测历史 → 推理时的错误会累积，长句子特别明显

4. **WER 是必要但不充分的指标**：WER 一样但用户体验可能差很多（关键词错了 vs 功能词错了）

5. **Whisper 不支持流式是硬限制**：很多团队想用 Whisper 做实时字幕，但它的架构天生需要等完整句子，这不是工程问题而是架构问题

## 💼 双视角面试题库

| 面试题 | 算法回答要点 | 产品回答要点 |
|---|---|---|
| "ASR 为什么用 CNN 做特征提取？" | 语音的关键特征具有局部性，CNN 的参数共享适合检测局部声学模式 | 理解即可，重点是局部特征提取的高效性 |
| "Whisper 和实时转写方案的区别？" | AED 需要完整输入，CTC/RNN-T 支持流式 | 离线 vs 实时，影响用户体验和产品设计 |
| "为什么 CTC 需要 blank 符号？" | 处理过渡帧 + 分隔重复字母，让折叠函数正确工作 | 了解即可 |
| "什么是 WER？能超过 100% 吗？" | $100×(S+I+D)/N$，可以超过 100% 因为有插入错误 | 是 ASR 的核心 KPI，选供应商时要对齐测试场景 |
| "HuBERT 如何减少对标注数据的依赖？" | 自监督预训练（掩码预测），用无标注音频学声学表示，微调只需少量数据 | 适合低资源语言和垂直领域快速适配 |
| "如何比较两个 ASR 系统的性能差异是否显著？" | MAPSSWE 检验（处理相邻词错误的相关性）| 避免用差 2% 这种未经统计检验的数字做产品决策 |

## 🔖 核心概念速查

| 术语 | 一句话定义 |
|---|---|
| **ASR / STT** | 声学波形 → 文字序列的映射函数 |
| **Log Mel Spectrum** | 128维/帧的声学特征，模拟人类听觉 |
| **Stride** | CNN kernel 每次移动的距离，控制序列压缩比 |
| **Cross-Attention** | 解码器动态关注编码器不同位置的机制 |
| **Teacher Forcing** | 训练时用真实标签而非预测结果作为下一步输入 |
| **HuBERT** | 自监督 ASR 预训练模型，掩码预测音素类 |
| **MFCC** | 39维声学特征，专门捕捉音素相关成分 |
| **CTC** | 每帧输出字符 + 折叠函数，支持流式，需外部 LM |
| **blank (ε)** | CTC 特有符号：无需转写帧 + 分隔重复字母 |
| **折叠函数 B** | 先合并重复字符，再删除 blank |
| **RNN-T** | CTC 升级版，加 Predictor 去除独立性假设 |
| **WER** | = 100×(S+I+D)/N，可超 100%，ASR 标准评估指标 |
| **MAPSSWE** | ASR 专用显著性检验，处理相邻词错误的相关性 |
| **Streaming** | 不等说完就开始输出，支持实时转写 |

## 🔖 核心公式速查卡

$$\text{WER} = 100 \times \frac{S+I+D}{N}$$

$$\text{CrossAttention}(Q,K,V) = \text{softmax}\!\left(\frac{QK^\top}{\sqrt{d_k}}\right) V$$

$$P_{CTC}(A|X) = \prod_{t=1}^{T} p(a_t|X) \quad \text{（独立性假设）}$$

$$P_{CTC}(Y|X) = \sum_{A \in B^{-1}(Y)} P(A|X) \quad \text{（正确做法：对所有对齐求和）}$$

$$L_{CTC} = -\log \sum_{A \in B^{-1}(Y)} \prod_{t=1}^{T} p(a_t|h_t) \quad \text{（训练损失）}$$

$$z_j = \sum_{i=-p}^{p} x_{j+i} \cdot w_{i+p} \quad \text{（带 Padding 的一维卷积）}$$
